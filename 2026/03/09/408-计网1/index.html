一、计算机网络的概念、组成与功能

1.1 基本概念

定义

计算机网络是指将地理上分散的、具有独立功能的多台计算机，通过 通信设备 和 传输介质 互联起来，在 网络软件 ( 协议、操作系统等 ) 的管理和协调下，实现 资源共享 和 信息传递 的系统.

关键词提炼 :

  • 独立功能 : 每台主机可单独运行，不依赖网络存在.
  • 互联 : 通过有线或无线介质连接.
  • 协议 : 通信双方必须遵守相同的规则.

1.2 组成结构

从组成角度划分

计算机网络由以下三部分组成 :

  1. 硬件
    • 主机 ( 端系统 ) : 服务器、PC、移动终端等，是网络的边缘部分.
    • 通信链路 : 双绞线、光纤、无线信道等传输介质.
    • 网络设备 : 路由器、交换机、集线器、调制解调器等，构成网络核心.
  2. 软件
    • 网络操作系统 ( 如 Linux、Windows Server )
    • 网络管理软件、网络应用程序 ( 浏览器、邮件客户端等 )
  3. 协议
    • 计算机网络的 核心，是通信双方必须共同遵守的规则集合.
    • 无协议，则网络中的通信无法进行.

从功能角度划分 ( 两大部分 )

部分 别称 组成 功能
边缘部分 资源子网 主机 ( 端系统 ) 信息处理，运行网络应用
核心部分 通信子网 路由器、链路等 数据交换与转发

核心部分的交换方式 :

交换方式 原理 特点
电路交换 建立专用物理通路 实时性好，但资源利用率低 ( 如电话网 )
报文交换 整个报文存储转发 延迟大，不适合实时通信
分组交换 报文拆分为分组，逐跳转发 灵活高效 ( 互联网采用此方式 )

1.3 三大功能

功能概述

  1. 数据通信 : 最基本的功能，在计算机之间传送各种信息 ( 文字、图像、视频等 ).

  2. 资源共享 : 共享软件 ( 程序 ) 、硬件 ( 打印机 ) 、数据 ( 数据库 ) ，提高资源利用率，降低成本.

  3. 分布式处理 : 将一个大型任务分布到多台计算机上协同完成，提高处理能力 ( 如云计算、MapReduce ).

此外还有 : 提高可靠性 ( 互为备份 ) 、促进集中管理等.

二、计算机网络的分类

2.1 按分布范围分类 ( 最常用 )

LAN / MAN / WAN

类型 全称 覆盖范围 典型距离 特点
LAN 局域网 一栋楼或一个校园 \(\le 10\ \text{km}\) 速率高、误码率低、归个人或单位所有
MAN 城域网 一个城市 \(\sim 50\ \text{km}\) 介于 LAN 与 WAN 之间
WAN 广域网 跨城市/国家/洲 \(\ge 100\ \text{km}\) 速率相对较低、误码率较高，需专业运营商

注意 : 互联网 ( Internet ) 是世界上规模最大的广域网，由无数个 LAN、MAN、WAN 互联而成，属于广域网范畴.

2.2 按传输技术分类

广播式网络 vs 点对点网络

类型 特征 典型示例
广播式网络 信道共享，所有节点收到同一帧，通过地址判断是否接收 以太网 ( 总线型 ) 、无线局域网
点对点网络 每条链路连接一对节点，分组需经过路由选择 广域网、Internet 骨干网

2.3 按拓扑结构分类

五种基本拓扑

拓扑结构 特点 优点 缺点
总线型 所有节点连到一条共享总线 布线少，扩展简单 冲突多，故障诊断困难
星型 所有节点连到中心交换机 管理方便，故障易隔离 中心节点单点故障
环型 节点依次连成环 结构简单，无冲突 ( 令牌环 ) 单点故障影响全网
网状型 节点间有多条路径 可靠性高，容错能力强 成本高，路由复杂
树型 层次化星型的扩展 扩展性好，管理方便 根节点故障影响全网

互联网的核心采用网状拓扑，保证高可靠性与冗余路径.

2.4 其他分类方式

按使用者 / 按交换技术 / 按传输介质

  • 按使用者 : 公用网 ( 由运营商建设，向公众开放 ) 、专用网 ( 特定机构内部使用，如军网、银行网 ).
  • 按交换技术 : 电路交换网、报文交换网、分组交换网.
  • 按传输介质 : 有线网 ( 双绞线、同轴电缆、光纤 ) 、无线网 ( WiFi、3G/4G/5G、卫星 ).

三、计算机网络的主要性能指标

3.1 速率与带宽

速率 ( Data Rate / Bit Rate )

速率即数据传输速率，指单位时间内传输的二进制位数，是网络最重要的性能指标之一.

  • 单位 : bps ( bit per second，比特/秒 ) ，常用 Kbps、Mbps、Gbps.
  • 注意 : 网络中 \(1\ \text{Kbps} = 10^3\ \text{bps}\) ( 十进制 ) ，存储中 \(1\ \text{KB} = 2^{10}\ \text{B}\) ( 二进制 ) ，两者不同！

带宽 ( Bandwidth )

在计算机网络中，带宽表示网络通信线路所能传送数据的 最高数据率，单位同为 bps.

  • 带宽是信道的 最大传输能力，是理论上限.
  • 在模拟通信领域，带宽指信号占用的频率范围 ( Hz ) ，两者意义不同但相互关联 ( Shannon 公式 ).

3.2 吞吐量

吞吐量 ( Throughput )

吞吐量指单位时间内通过某网络 ( 或接口 ) 的 实际数据量.

\[\text{吞吐量} \le \text{带宽 ( 信道容量 ) }\]

  • 吞吐量受网络拥塞、误码率、协议开销等因素影响，通常小于带宽.
  • 是衡量网络 实际工作性能 的重要指标.

示例 : 一条 100 Mbps 的以太网，实际吞吐量可能只有 60~70 Mbps，原因包括帧间隔、冲突重传、协议开销等.

3.3 时延 ( 重点 )

时延的四个组成部分

总时延 = 发送时延 + 传播时延 + 处理时延 + 排队时延

\[t_{total} = t_s + t_p + t_{proc} + t_q\]

时延类型 产生位置 公式 影响因素
发送时延 \(t_s\) 发送端 ( 网卡发送过程 ) \(t_s = \dfrac{L}{B}\) ( \(L\) : 数据长度 bit，\(B\) : 信道带宽 ) 数据长度、带宽
传播时延 \(t_p\) 传输介质中 \(t_p = \dfrac{d}{v}\) ( \(d\) : 信道长度，\(v\) : 传播速度 ) 信道长度、介质类型
处理时延 \(t_{proc}\) 中间节点 ( 路由器等 ) — 节点处理能力
排队时延 \(t_q\) 中间节点队列 — 网络拥塞程度

传播速度 : 光纤/铜线中约 \(2 \times 10^8\ \text{m/s}\)，自由空间 ( 无线 ) 约 \(3 \times 10^8\ \text{m/s}\).

易混淆 : 发送时延 vs 传播时延

发送时延 传播时延
本质 将数据”推上”信道所需时间 数据在信道中”飞行”的时间
与数据量关系 成正比 无关
与信道长度关系 无关 成正比
与带宽关系 成反比 无关

举例 : 发送 \(10^6\) bit 的数据，信道带宽 \(1\ \text{Mbps}\)，信道长度 \(1000\ \text{km}\) ( 光纤 ).

  • 发送时延 : \(\dfrac{10^6}{10^6} = 1\ \text{s}\)
  • 传播时延 : \(\dfrac{10^6}{2 \times 10^8} = 5 \times 10^{-3}\ \text{s} = 5\ \text{ms}\)
  • 此时发送时延 远大于 传播时延.

若信道带宽升为 \(1\ \text{Gbps}\)，发送时延变为 \(1\ \text{ms}\)，传播时延仍为 \(5\ \text{ms}\)，此时传播时延成为瓶颈.

3.4 时延带宽积

时延带宽积 ( Bandwidth-Delay Product )

\[\text{时延带宽积} = \text{传播时延} \times \text{信道带宽} = t_p \times B\]

  • 单位 : bit
  • 物理意义 : 某段时刻链路上”正在传播中” ( 已离开发送端但尚未到达接收端 ) 的比特数，又称 以比特为单位的链路长度.
  • 反映链路的”容量”，在流量控制、拥塞控制设计中有重要意义.

示例 : 带宽 \(1\ \text{Gbps}\)，传播时延 \(20\ \text{ms}\). \[\text{时延带宽积} = 1 \times 10^9 \times 0.02 = 2 \times 10^7\ \text{bit} = 20\ \text{Mbit}\] 即链路上同时”飞行”着 2000 万个比特！

3.5 往返时间 RTT

RTT ( Round-Trip Time )

RTT 是指从 发送方发出第一个比特 到 收到接收方的确认 所经历的时间.

\[\text{RTT} \approx 2 \times t_p + \text{处理时延}\]

  • 在 TCP 协议中，RTT 是衡量网络拥塞的重要依据，用于动态计算超时重传时间 ( RTO ).
  • 实际 RTT 通过加权平均 ( EWMA ) 估计 : \(\text{EstimatedRTT} = (1-\alpha)\cdot\text{EstimatedRTT} + \alpha\cdot\text{SampleRTT}\)，推荐 \(\alpha = 0.125\).

3.6 利用率

信道利用率与网络利用率

\[U = \frac{\text{有数据通过的时间}}{\text{总时间}} \quad \text{或} \quad U = \frac{\text{吞吐量}}{\text{带宽}}\]

非常重要的结论 : 利用率并非越高越好！

根据排队论，当信道利用率 \(U\) 增大时，时延 \(D\) 按如下规律急剧增大 :

\[D = \frac{D_0}{1-U}\]

其中 \(D_0\) 为网络空闲时的时延. 当 \(U \to 1\) 时，\(D \to \infty\).

推论 : 为保证较低的时延，一般将信道利用率控制在 50% 以下. 过高的利用率会导致严重拥塞.

性能指标对比记忆表

指标 描述对象 关键公式
速率/带宽 信道 最大 传输能力 \(B\) ( bps )
吞吐量 网络 实际 通过量 \(\le B\)
发送时延 数据”装上”信道 \(L/B\)
传播时延 信号”穿越”信道 \(d/v\)
时延带宽积 链路上飞行的比特数 \(t_p \times B\)
RTT 一来一回总时延 \(\approx 2t_p\)
利用率 信道忙碌程度 \(\le 50\%\) 为宜

四、计算机网络体系结构

4.1 分层结构的思想

为什么要分层?

计算机网络是极其复杂的系统，面临的挑战包括 :

  • 不同硬件、操作系统、应用程序之间的异构性
  • 数据传输的可靠性、安全性保障
  • 网络规模的可扩展性

分层的基本原则 :

  1. 每层完成特定的功能，层与层之间通过接口通信.
  2. 上层使用下层提供的服务，无需了解下层的实现细节 ( 封装性 ).
  3. 同层实体通过协议通信 ( 逻辑通信 ) ，实际数据通过物理介质传输.

分层带来的好处 : 各层可独立设计、实现和修改，降低了系统复杂度 ( 模块化设计 ).

4.2 协议、接口、服务

三个核心概念

1. 协议 ( Protocol )

协议是 控制两个对等实体进行通信的规则集合，规定了通信双方在通信时 必须遵守的格式和时序.

协议由三要素组成 :

  • 语法 ( Syntax ) : 数据的格式、编码与信号电平 ( 数据的结构是什么样的 ).
  • 语义 ( Semantics ) : 控制信息的含义，需要完成的动作及应做出的响应 ( 每个字段代表什么含义 ).
  • 时序 ( Timing ) : 事件的顺序与速度匹配 ( 何时发送、如何应答 ).

2. 接口 ( Interface )

接口是 同一节点内相邻两层之间的边界，定义了下层向上层提供的操作与服务 ( 服务访问点 SAP, Service Access Point ).

3. 服务 ( Service )

服务是 低层向高层提供的功能，分为 : | 服务类型 | 描述 | 典型协议 | | :— | :— | :— | | 面向连接服务 | 通信前先建立连接，可靠但开销大 | TCP | | 无连接服务 | 直接发送，不建立连接，不可靠但高效 | UDP |

协议 接口 服务
方向 水平 ( 同层对等实体间 ) 垂直 ( 相邻层间 ) 垂直 ( 下层→上层 )
对象 对等实体 相邻层 上层用户

服务与协议的区别

  • 协议是”水平的” : 规定了同层对等实体如何通信，是对等层之间的”约定”.
  • 服务是”垂直的” : 描述了某层能为其上层提供哪些功能，通过服务访问点 ( SAP ) 向上层提供.
  • 协议的实现 借助下层提供的服务，同时协议的运行又 为上层提供新的服务.

举例 : 运输层协议 ( TCP ) 使用网络层 ( IP ) 提供的服务，同时向应用层提供可靠传输服务.

4.3 ISO/OSI 参考模型

OSI 七层模型 ( 从高到低 )

层次 名称 主要功能 典型协议/设备
第 7 层 应用层 为用户提供网络服务接口 HTTP, FTP, SMTP, DNS, Telnet
第 6 层 表示层 数据格式转换、加密解密、压缩解压 JPEG, MPEG, SSL ( 部分 )
第 5 层 会话层 建立、管理和终止会话，提供同步 NetBIOS, RPC
第 4 层 传输层 端到端可靠传输，流量控制，错误恢复 TCP, UDP
第 3 层 网络层 逻辑寻址，路由选择，分组转发 IP, ICMP, ARP ( 争议 ) , 路由器
第 2 层 数据链路层 帧同步，差错控制，流量控制，MAC寻址 Ethernet, PPP, 交换机, 网桥
第 1 层 物理层 比特流传输，定义物理接口和信号 集线器, 中继器, 网线, 光纤

数据单元名称 :

层次 数据单元 ( PDU )
应用层/表示层/会话层 报文 ( Message )
传输层 报文段 ( Segment )——TCP / 用户数据报 ( Datagram )——UDP
网络层 数据报 / 分组 ( Packet )
数据链路层 帧 ( Frame )
物理层 比特 ( Bit )

各层功能详解

1. 物理层

  • 规定传输介质的物理、电气、功能和规程特性.
  • 主要任务 : 将比特流转化为物理信号 ( 电信号、光信号、无线电波 ) 在介质上传输.
  • 关键设备 : 集线器 ( Hub ) 、中继器 ( 工作在物理层，不识别 MAC 地址 ).

2. 数据链路层

  • 将网络层传来的数据封装成 帧，在 相邻节点 之间的链路上传输.
  • 功能 : 成帧 ( 帧定界 ) 、差错控制 ( CRC ) 、流量控制、介质访问控制 ( MAC ).
  • 关键设备 : 网桥、交换机 ( 根据 MAC 地址转发，工作在数据链路层 ).

3. 网络层

  • 将 分组 从源端传送到目的端，可能跨越多个网络 ( 异构网络互连 ).
  • 功能 : 路由选择 ( 选择最优路径 ) 、拥塞控制、网际互连.
  • 关键设备 : 路由器 ( 根据 IP 地址转发，工作在网络层 ).

4. 传输层

  • 实现 端到端 ( 进程到进程 ) 的可靠数据传输，屏蔽下层网络的差异.
  • 功能 : 复用/分用 ( 端口号 ) 、流量控制、差错检测与恢复、连接管理.
  • 关键协议 : TCP ( 面向连接，可靠 ) 、UDP ( 无连接，不可靠但高效 ).

5. 会话层、表示层、应用层 ( 在 TCP/IP 中合并为应用层 )

  • 会话层 : 管理对话 ( 半双工/全双工 ) ，提供同步 ( 检查点机制 ).
  • 表示层 : 处理数据的表示、编码格式 ( ASCII/EBCDIC ) 、加密、压缩.
  • 应用层 : 直接为用户应用提供服务，是用户与网络的接口.

4.4 TCP/IP 模型

TCP/IP 四层模型

层次 名称 对应 OSI 层 典型协议
第 4 层 应用层 应用层+表示层+会话层 HTTP, HTTPS, FTP, SMTP, POP3, DNS, DHCP, Telnet, SSH
第 3 层 传输层 传输层 TCP, UDP
第 2 层 网际层 ( 网络互联层 ) 网络层 IP, ICMP, ARP, RARP, OSPF, BGP
第 1 层 网络接口层 数据链路层+物理层 Ethernet, WiFi(802.11), PPP

TCP/IP 的特点 :

  • 实用性强，已成为事实上的工业标准.
  • 网际层的 IP 协议是整个体系结构的核心，提供 无连接、不可靠 的”尽力而为”服务.
  • 可靠性由传输层 ( TCP ) 保证，而非网络层.

4.5 OSI 与 TCP/IP 的对比 ( 考研重点 )

五层参考模型 ( 考研常用 )

考研中通常使用综合 OSI 与 TCP/IP 的 五层参考模型 ( 自顶向下 ) :

层次 名称 核心任务
5 应用层 网络应用与协议 ( HTTP, DNS, FTP… )
4 传输层 进程间通信 ( TCP, UDP )
3 网络层 主机间通信，路由转发 ( IP )
2 数据链路层 直连节点间帧传输 ( Ethernet )
1 物理层 比特流传输 ( 电信号/光信号 )

OSI 与 TCP/IP 对比

比较维度 OSI 参考模型 TCP/IP 模型
层数 7 层 4 层 ( 或 5 层 )
制定背景 理论先行，国际标准化组织 ( ISO ) 制定 实践先行，ARPANET 演化而来
适用场景 理论研究、理解网络功能 实际工程应用，互联网标准
网络层服务 面向连接 + 无连接 仅无连接 ( IP )
传输层服务 面向连接 面向连接 ( TCP ) + 无连接 ( UDP )
普及程度 未被广泛应用 互联网实际采用
缺点 层次过多，部分层功能不明确 网络接口层定义不明确，无法描述物理网络

发送方主机 ( 应用层 \(\to\) 物理层 ) 各层依次添加首部 ( Header ) :

接收方则逆向解封装，每层剥掉对应首部并传给上层.

中间路由器 只需处理到网络层 ( 查看 IP 头进行路由转发 ) ，不需要了解传输层及以上内容，这体现了分层的优越性.

五、综合真题解析 ( Selected Problems )

真题 1 ( 性能指标计算 )

一个 \(10\ \text{Mbps}\) 的以太网，发送 \(1000\ \text{B}\) 的帧到距离 \(1\ \text{km}\) 处的目的主机 ( 信号传播速度 \(2\times 10^8\ \text{m/s}\) ) ，求总时延.

解 : 数据长度 : \(L = 1000 \times 8 = 8000\ \text{bit}\)

发送时延 : \[t_s = \frac{L}{B} = \frac{8000}{10 \times 10^6} = 0.8 \times 10^{-3}\ \text{s} = 0.8\ \text{ms}\]

传播时延 : \[t_p = \frac{d}{v} = \frac{1000}{2 \times 10^8} = 5 \times 10^{-6}\ \text{s} = 0.005\ \text{ms}\]

忽略处理时延和排队时延，总时延 : \[t = t_s + t_p = 0.8\ \text{ms} + 0.005\ \text{ms} \approx 0.805\ \text{ms}\]

结论 : 在这种情况下，发送时延远大于传播时延，提高带宽可显著减少总时延.

真题 2 ( 时延带宽积应用 )

一条带宽 \(100\ \text{Mbps}\) 的光纤链路，传播时延为 \(20\ \text{ms}\). 发送方连续发送数据，问在收到第一个确认之前，可以发出多少比特的数据?

解 : RTT 近似为 \(2 \times t_p = 40\ \text{ms}\). 在 RTT 内发送方可发出的数据量 :

\[\text{数据量} = \text{带宽} \times \text{RTT} = 100 \times 10^6 \times 40 \times 10^{-3} = 4 \times 10^6\ \text{bit} = 4\ \text{Mbit}\]

这就是为何 TCP 的窗口大小 ( 发送缓冲区 ) 必须足够大，才能充分利用高带宽长时延链路——这类链路被称为“长肥管道” ( Long Fat Network ) .

真题 3 ( OSI 层次判断 )

判断以下设备/协议分别工作在哪一层? (1) 集线器 ( Hub ) 　(2) 交换机 ( Switch ) 　(3) 路由器 ( Router ) (4) 网关 ( Gateway ) 　(5) ARP 协议　(6) ICMP 协议

解 :

设备/协议 工作层次 说明
集线器 Hub 物理层 纯信号中继，不识别地址，广播式转发
交换机 Switch 数据链路层 根据 MAC 地址表转发帧
路由器 Router 网络层 根据 IP 路由表转发分组
网关 Gateway 传输层及以上 协议转换，工作在高层
ARP 协议 网络层 ( 有争议 ) 功能上介于网络层与数据链路层之间，408 中归属网络层
ICMP 协议 网络层 IP 的辅助协议，封装在 IP 数据报中传输

真题 4 ( 分层封装分析 )

主机 A 向主机 B 发送 HTTP 请求报文，中间经过一台路由器，试分析各设备处理的层次和内容.

解 :

主机 A ( 发送方 ) :

  • 应用层生成 HTTP 请求，交给传输层.
  • 传输层 ( TCP ) 添加 TCP 首部 ( 源/目端口、序号等 ) ，形成报文段.
  • 网络层 ( IP ) 添加 IP 首部 ( 源/目 IP 地址 ) ，形成 IP 数据报.
  • 数据链路层添加以太网帧头帧尾 ( 源/目 MAC 地址 ) ，形成帧.
  • 物理层将帧转为比特流发送.

路由器 ( 中间节点 ) :

  • 物理层接收比特流还原为帧.
  • 数据链路层 剥去帧头帧尾，验证 CRC，取出 IP 数据报.
  • 网络层查看目标 IP 地址，查路由表，选择输出端口.
  • 数据链路层 重新封装新的帧 ( 更换 MAC 地址 ) ，物理层发送.
  • 注意 : 路由器不处理传输层及以上内容.

主机 B ( 接收方 ) :

  • 逐层解封装，每层剥掉对应首部，直到应用层还原 HTTP 请求.

真题 5 ( 协议概念辨析 )

下列说法是否正确? 请逐一分析.

  1. 两台计算机通过网络通信时，数据是在对等层之间实际传输的.

  2. 协议是相邻层之间通信的规则，接口是同层对等实体之间通信的规则.

  3. OSI 参考模型是互联网实际采用的标准.

  4. 增大信道利用率一定能提高网络性能.

解 :

  1. 错误. 对等层之间是 逻辑通信 ( 虚通信 ) ，实际数据传输只发生在物理层 ( 通过物理介质 ). 高层的通信都要向下传递，最终由物理层完成实际传输.

  2. 错误. 恰好 说反了. 协议是 同层对等实体 之间通信的规则 ( 水平方向 ); 接口是 相邻层 之间的边界 ( 垂直方向 ).

  3. 错误. OSI 参考模型是国际标准，但在实际互联网中 广泛采用的是 TCP/IP 模型. OSI 更多用于理论学习和理解，TCP/IP 才是工业标准.

  4. 错误. 利用率并非越高越好. 根据排队论，当信道利用率 \(U\) 过高时，网络时延会急剧增大 ( \(D = D_0/(1-U)\) ) ，反而 降低网络性能. 实际运营中一般将利用率控制在 50% 以下.
